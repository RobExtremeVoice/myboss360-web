# Intelligence Architecture — Memory Engine & Learning Engine

## Overview

MyBoss360's intelligence layer is composed of two complementary systems that work beneath the UI to make the platform smarter over time: the **Memory Engine** and the **Learning Engine**. Neither system calls an external LLM directly — they are the structured data layer that a future Executive AI layer will read from and write to.

---

## Memory Engine

**Purpose:** Store durable, structured business memory that persists across sessions and surfaces as context when the Executive AI needs to generate insights, summaries, or decisions.

### What it stores

| Memory type | Description |
|---|---|
| `user_preference` | How a specific user likes to work (communication style, reporting cadence, etc.) |
| `org_goal` | Strategic objectives at the organization level (OKRs, annual targets) |
| `workspace_context` | Workspace-level operational context (team composition, current quarter priorities) |
| `decision` | Important business decisions that were made and why |
| `meeting_summary` | Structured summaries extracted from meeting notes |
| `observation` | Business observations logged manually or by the system |
| `executive_note` | Free-form notes written by an executive for future recall |
| `ai_insight` | Insights generated by the AI layer (time-limited, marked with confidence) |
| `historical_recommendation` | Recommendations surfaced in the past |
| `accepted_recommendation` | Recommendations that were explicitly accepted and acted on |
| `rejected_recommendation` | Recommendations that were explicitly rejected (feeds the learning loop) |

### Key design decisions

- **Workspace + org isolation:** every memory is scoped to both a `workspace_id` and `organization_id`. RLS policies enforce that only workspace members can read or write memories.
- **Entity linking:** memories can be pinned to a specific entity (`entity_type` + `entity_id`), enabling context retrieval per deal, contact, or project.
- **Confidence scores:** AI-generated memories carry a `confidence` field (0.00–1.00). Future AI layers can filter by confidence to avoid surfacing low-quality memories.
- **Expiration:** time-sensitive memory types (e.g., `ai_insight`) carry an `expires_at` timestamp. A scheduled job or the service layer can prune or archive them.
- **Audit trail:** `memory_events` records every access, update, and expiration. This enables the learning engine to see which memories were useful and which were ignored.

### Files

```
types/memory.ts                       — Application-level TypeScript types
config/memory.ts                      — Type labels, expiration defaults, groupings
repositories/memory/memories.ts       — Supabase CRUD for memories + memory_events
services/memory/memory-service.ts     — createMemory, listMemories, recordMemoryEvent,
                                        getExecutiveContext
```

### Key service functions

| Function | Description |
|---|---|
| `createMemory(input)` | Persists a memory and records a `created` memory event |
| `listMemories(workspaceId, options)` | Lists memories with optional type, entity, and user filters |
| `recordMemoryEvent(memoryId, eventType, actorId)` | Appends an event to a memory's audit trail |
| `getExecutiveContext(workspaceId, organizationId)` | Assembles the full executive context snapshot: org goals, decisions, workspace context, AI insights |

## Learning Engine

**Purpose:** Detect recurring patterns in operational data (deals, tasks, activities) and surface actionable recommendations — without requiring LLM calls.

### What it tracks

**Signals** are individual data points that indicate something worth watching:

| Signal type | Triggered when |
|---|---|
| `deal_risk` | A deal is approaching its close date with low probability or no recent activity |
| `follow_up_delay` | A contact or lead has not been followed up within the threshold window |
| `task_delay` | A task is past its due date |
| `customer_health` | A company's health score drops or engagement decreases |
| `sales_pattern` | A recurring sales behaviour is detected (e.g. deals consistently stalling at proposal) |
| `recurring_bottleneck` | The same type of blocker appears repeatedly across projects or deals |
| `performance_trend` | A metric (revenue, close rate, task completion) is trending in a notable direction |
| `recommended_action` | A specific action is generated from signal data |

**Patterns** are formed when the same signal type fires repeatedly. The `detectPatterns()` function groups unresolved signals, counts occurrences, and computes a confidence score. Once a pattern crosses the minimum occurrence and confidence thresholds, it is written to `learning_patterns`.

**Recommendations** are actionable items surfaced to users based on signals or patterns. They carry a priority (`low`, `medium`, `high`, `critical`) and a status lifecycle: `pending → accepted | rejected | dismissed | expired`.

**Feedback** closes the loop: when a user accepts or rejects a recommendation, `recommendation_feedback` captures their response and moves the recommendation to the corresponding terminal status. Over time, rejected recommendations feed back into pattern detection to reduce false positives.

### Key design decisions

- **Heuristic-first:** pattern detection uses frequency counting and severity weighting, not LLM inference. This keeps the system fast, deterministic, and cheap to run.
- **Upsert semantics:** `detectPatterns()` increments an existing pattern's `occurrences` and updates its `confidence` rather than creating duplicates.
- **Workspace isolation:** all signals and patterns are workspace-scoped with RLS enforcement.
- **Configurable thresholds:** `config/learning.ts` exposes all numeric thresholds (`minPatternOccurrences`, `followUpDelayThresholdDays`, etc.) in one place so they can be tuned without touching service logic.

### Files

```
types/learning.ts                              — Application-level TypeScript types
config/learning.ts                             — Signal/pattern labels, thresholds, weights
repositories/learning/signals.ts              — CRUD for learning_signals
repositories/learning/patterns.ts             — CRUD for learning_patterns
repositories/learning/recommendations.ts      — CRUD for recommendations + feedback
services/learning/learning-service.ts         — createLearningSignal, detectPatterns,
                                                createRecommendation,
                                                recordRecommendationFeedback,
                                                listRecommendations, listPatterns
```

### Key service functions

| Function | Description |
|---|---|
| `createLearningSignal(input)` | Records a new signal from any part of the system |
| `detectPatterns(workspaceId, orgId)` | Groups unresolved signals into patterns; upserts pattern records |
| `createRecommendation(input)` | Surfaces a recommendation linked to a pattern or signal |
| `recordRecommendationFeedback(id, userId, action)` | Records user feedback and transitions the recommendation status |
| `listRecommendations(workspaceId, options)` | Lists pending (or filtered) recommendations for a workspace |
| `listPatterns(workspaceId, options)` | Lists detected patterns above the confidence threshold |

## How Memory and Learning differ from LLM providers

| Dimension | Memory Engine | Learning Engine | LLM Provider (future) |
|---|---|---|---|
| **Source of truth** | Structured DB records | Signal aggregation | External API call |
| **Latency** | Milliseconds | Milliseconds | Seconds |
| **Cost** | Database reads | Database reads | Per-token billing |
| **Determinism** | Fully deterministic | Fully deterministic | Stochastic |
| **Context window** | Unlimited (filtered) | Unlimited (aggregated) | Limited by model |
| **Primary role** | Stores what happened | Detects what's recurring | Generates language |

The Memory Engine and Learning Engine are the **context providers**. When a future Executive AI layer calls an LLM, it will pass the output of `getExecutiveContext()` plus relevant patterns and recommendations as grounded context — ensuring the model's responses are rooted in real business data rather than hallucinated generalities.

## How the Executive AI will use these systems

```
User request
    ↓
Executive AI (future)
    ├── getExecutiveContext(workspaceId, orgId)     ← Memory Engine
    ├── listPatterns(workspaceId)                   ← Learning Engine
    ├── listRecommendations(workspaceId)            ← Learning Engine
    └── [assemble prompt with structured context]
         ↓
    LLM call (future)
         ↓
    [parse response]
         ├── createMemory({ type: 'ai_insight', ... })    ← writes back to Memory Engine
         └── createRecommendation({ ... })                ← writes to Learning Engine
```

This loop means that every AI interaction improves the context available to the next one.

## Future MCP / tool integration

The service functions are designed to be wrapped as MCP tools with minimal adaptation:

- `createMemory` → `store_memory` tool
- `getExecutiveContext` → `get_executive_context` tool  
- `createLearningSignal` → `record_signal` tool
- `detectPatterns` → `detect_patterns` tool (runs on a schedule or on-demand)
- `createRecommendation` → `surface_recommendation` tool
- `recordRecommendationFeedback` → `submit_feedback` tool

Each function already accepts plain typed inputs and returns plain typed outputs — no MCP-specific changes to the core logic will be needed.
