# v1.1 — Knowledge Engine (RAG Foundation)

**Status:** Shipped  
**Theme:** Permanent knowledge layer and document intelligence infrastructure  
**Commit range:** `51f78f8` → `2ef3356`

---

## Objective

v1.1 establishes the foundational knowledge layer for the MyBoss360 platform. Every document, policy, SOP, meeting note, and executive decision can now be stored, version-controlled, and searched. The architecture is built for Retrieval-Augmented Generation (RAG) from the ground up — semantic and hybrid search interfaces are defined and wired, ready to be activated with real embeddings in v1.3.

This release is "architecture-first": no OpenAI embeddings yet, but every piece required for full RAG is in place. Sprint 20 (v1.3) drops in embeddings and activates the semantic layer without requiring schema changes.

## Major Milestones

### 1. Database Schema (9 tables)

| Table | Purpose |
|---|---|
| `knowledge_collections` | Logical groupings of documents |
| `knowledge_sources` | Origin system registry |
| `knowledge_documents` | Core entity: typed, categorized, versioned, soft-deletable |
| `knowledge_chunks` | RAG fragments with `embedding_id` reserved for pgvector |
| `knowledge_tags` | Workspace-scoped taxonomy |
| `document_tags` | Document ↔ tag junction |
| `knowledge_links` | Typed doc-to-doc relationships |
| `document_versions` | Immutable snapshot before each update |
| `document_permissions` | Per-document ACL (user or role) |

Full RLS on all 9 tables. Trigram GIN indexes on `title` and `content` power keyword search. `pg_trgm` extension enabled.

### 2. Document Pipeline

```
Raw content (text)
    │
    ▼ parseDocument()
Normalized content + word count + character count
    │
    ▼ chunkDocument()
Chunks (strategy: paragraph | sentence | fixed_size | semantic*)
    │         * semantic falls back to paragraph until v1.3
    ▼
Stored in knowledge_chunks with estimated token counts
```

Token estimation: `Math.ceil(chars / 4)` — calibrated for English text, no external dependencies.

### 3. Search Service

Three modes sharing the `SearchQuery` / `SearchResponse` contract:

| Mode | Implementation | Status |
|---|---|---|
| `keyword` | Supabase `ilike` + trigram GIN index | ✅ Live |
| `semantic` | Mock (falls back to keyword) | ⏳ Activates v1.3 |
| `hybrid` | Mock (falls back to keyword, RRF scoring planned) | ⏳ Activates v1.3 |

The contract is fixed: v1.3 will replace the mock implementations without changing any caller code.

### 4. Knowledge Service

Full CRUD orchestrator:
- `createDocument` — runs pipeline, stores chunks, applies tags
- `updateDocument` — saves immutable version snapshot, then applies update, re-chunks if content changed
- `deleteDocument` — soft delete (sets `deleted_at`)
- `ensureDefaultCollection` — idempotent default collection creation

### 5. REST API

`GET /api/knowledge` — list | search | collections | tags  
`POST /api/knowledge` — create document | collection | tag  
`PATCH /api/knowledge` — update document  
`DELETE /api/knowledge` — soft-delete document

All routes: Supabase SSR auth → workspace resolution → service call → try/catch JSON 500.

### 6. Architecture Documentation

Four documents in `docs/architecture/`:
- `README.md` — navigation and Mermaid overview
- `system-architecture.md` — 12-section platform guide
- `database-erd.md` — all 27 tables, columns, RLS, indexes

### 7. Security Fix

PostgREST filter injection (MEDIUM) in `search-service.ts`: user query was interpolated directly into `.or()` filter string. Fixed by stripping `,().:*` metacharacters via `sanitizePostgrestValue()` before interpolation.

## Architecture

```
app/api/knowledge/route.ts
    │
    └─ createKnowledgeService(supabase)
            │
            ├─ createKnowledgeDocumentsRepository(db)
            ├─ createKnowledgeChunksRepository(db)
            ├─ createKnowledgeCollectionsRepository(db)
            └─ createKnowledgeTagsRepository(db)

    └─ createKnowledgeSearchService(supabase)
            └─ keywordSearch → Supabase ilike + GIN index

Document Write Flow:
    POST /api/knowledge
        └─ runDocumentPipeline(rawContent)
                ├─ parseDocument()     → ParseResult
                └─ chunkDocument()     → ChunkOutput
        └─ docsRepo.create()           → knowledge_documents
        └─ chunksRepo.createMany()     → knowledge_chunks
        └─ docsRepo.setTags()          → document_tags
```

## What's Next (v1.3)

To activate full RAG:
1. Enable `pgvector` extension in Supabase
2. Add `vector(1536)` column to `knowledge_chunks`
3. Wire OpenAI `text-embedding-3-small` to embed chunks on create/update
4. Implement `semanticSearch` (replace mock): embed query → `ORDER BY embedding <-> $1`
5. Implement `hybridSearch` with RRF scoring
6. Inject top-K chunks into AI assistant system prompt per user message
