## Objective

Before v1.0, MyBoss360 had no formal onboarding flow. New users would log in and land directly on the dashboard with no context, no workspace, and no guidance. The platform had no concept of "first run" — provisioning had to be done manually by an administrator.

v1.0 introduced a premium 8-step onboarding wizard and an automatic provisioning engine that creates a fully configured workspace on the user's first login — with zero manual intervention.

---

## Major Milestones

### 1. Onboarding Schema

A new `onboarding_state` table tracks the current step and completion status per user. The state machine has 8 steps:

```
welcome → company_name → industry → company_size → country → currency → business_goals → finish
```

### 2. Workspace Auto-Provisioning

On first login, the provisioning service atomically creates:
- Organization record
- Workspace record (linked to organization)
- Default "Executive" role
- Org-level membership (user → organization)
- Workspace-level membership (user → workspace)

If any step fails, a cleanup handler deletes the partially created organization — ensuring the database is never left in a partial state.

### 3. 8-Step Onboarding Wizard

A premium React wizard (`components/onboarding/OnboardingWizard.tsx`) with:
- Animated step transitions
- Progress indicator
- Form validation per step
- Server actions (PATCH `/api/onboarding`) to advance state
- Automatic redirect to `/dashboard` on completion

### 4. Dashboard Gate

The dashboard layout now checks onboarding completion via the service-role client on every request. Incomplete users are redirected to `/onboarding` before any dashboard content is rendered.

### 5. API Hardening

- Step whitelist guard: PATCH handler validates the submitted step against `ONBOARDING_STEPS`; returns HTTP 400 for unknown step names (previously would silently reset to `welcome`)
- All handlers (GET, POST, PATCH) wrapped in try/catch returning structured JSON 500

### 6. AI Error Handling Fix

- AI messages route returns HTTP 503 (not 404) when workspace is missing
- Chat UI renders the error in the empty state with clear guidance

## Architecture

```
First Login
    │
    ▼
app/(onboarding)/onboarding/page.tsx  ← Server Component
    │
    ├─ getUser() → redirect to /login if not authenticated
    ├─ getStatus(userId) → check onboarding_state
    │
    ├─ [status = not_started]
    │   └─ provisionWorkspace(userId)   ← Admin client (service-role)
    │       ├─ createOrganization
    │       ├─ createWorkspace
    │       ├─ createRole
    │       ├─ createOrgMembership
    │       └─ createWorkspaceMembership
    │
    ├─ [status = in_progress]
    │   └─ render OnboardingWizard (initialStep, completedSteps)
    │
    └─ [status = complete]
        └─ redirect('/dashboard')
```

## Security Notes

- `SUPABASE_SERVICE_ROLE_KEY` is used exclusively in the admin client; it is never prefixed `NEXT_PUBLIC_` and never sent to the browser
- The provisioning service is called only from server-side code (Server Component, API routes)
- Dashboard layout gate uses the service-role client to bypass RLS — this ensures the gate cannot be bypassed by an unauthenticated request forging cookies
